# 生产部署与配置收口手册

这份手册用来把当前项目从“本地联调可用”收口到“生产部署有明确基线”。

它聚焦四件事：

- 配置分层
- 部署角色
- 上线前检查
- 回滚与排障入口

## 1. 推荐配置分层

建议把生产配置拆成三层：

1. 基础层  
   使用 [`prod.base.env.example`](/home/fmy/NexusCore/env/prod.base.env.example)
2. Broker 层  
   三选一：
   - [`prod.broker.rabbitmq.env.example`](/home/fmy/NexusCore/env/prod.broker.rabbitmq.env.example)
   - [`prod.broker.nats.env.example`](/home/fmy/NexusCore/env/prod.broker.nats.env.example)
   - [`prod.broker.redis_streams.env.example`](/home/fmy/NexusCore/env/prod.broker.redis_streams.env.example)
3. 告警层  
   使用 [`prod.alerting.env.example`](/home/fmy/NexusCore/env/prod.alerting.env.example)

不要把所有变量堆在一个文件里，这样更方便：

- 按 broker 切换
- 按 role 复用
- 做配置审计

## 2. 生产角色建议

推荐至少按下面方式拆角色：

- `gateway`
  - 对外 HTTP 入口
  - 暴露 `/metrics`
- `runtime-worker`
  - 执行 broker reserve / judge / ack
  - 暴露 `/metrics` 和 `/api/v1/runtime/broker`
- `embedded`
  - 只建议在单机、小规模或临时环境使用
  - 正式环境更推荐拆成 `gateway + runtime-worker`

## 3. 基础部署建议

### 3.1 应用

- 使用 `NEXUS_ENV=prod`
- 使用 `NEXUS_LOG_FORMAT=json`
- `NEXUS_RUNTIME_SECCOMP_MODE=kill`
- `NEXUS_OJ_REPOSITORY=postgres`
- `NEXUS_BIND_ADDR=0.0.0.0:8080`

### 3.2 依赖

- PostgreSQL 使用持久化卷和定期备份
- Redis 至少启用持久化策略
- RabbitMQ / NATS / Redis Streams 三者只选一个作为主 broker
- Prometheus / Grafana / Alertmanager 使用独立卷

### 3.3 监控与告警

- Prometheus 必须能抓到应用 `/metrics`
- Grafana dashboard 必须已 provision
- Alertmanager 必须接通至少一个真实通知通道
- `nexus-alert-relay` 审计日志目录必须可写

## 4. 上线前检查

### 4.1 加载配置

示例：

```bash
set -a
source env/prod.base.env.example
source env/prod.broker.rabbitmq.env.example
source env/prod.alerting.env.example
set +a
```

### 4.2 跑配置检查

```bash
./scripts/check_deploy_config.sh
```

这个脚本会检查：

- role 是否有效
- broker 是否有效
- repository 是否有效
- 对应 broker 的必需变量是否齐全
- PostgreSQL 必需变量是否齐全
- 生产环境常见风险项是否仍停留在 dev/log/memory

### 4.3 跑 P0 验证

按顺序执行：

1. [`P0_生产稳定性与运维闭环_Runbook.md`](/home/fmy/NexusCore/P0_生产稳定性与运维闭环_Runbook.md)
2. `./scripts/verify_alert_pipeline.sh`
3. `./scripts/run_broker_failure_drill.sh <broker> runtime-worker`
4. `./scripts/smoke_runtime_matrix.sh`

## 5. 启动方式

### 5.1 Gateway

```bash
export NEXUS_PROCESS_ROLE=gateway
cargo run -p nexus-app
```

### 5.2 Runtime Worker

```bash
export NEXUS_PROCESS_ROLE=runtime-worker
cargo run -p nexus-app
```

### 5.3 Embedded

```bash
export NEXUS_PROCESS_ROLE=embedded
cargo run -p nexus-app
```

## 6. 回滚建议

如果 broker 或 runtime 升级后出现异常，优先按下面顺序回滚：

1. 回滚应用版本
2. 保持 broker 数据卷不删
3. 保留 Prometheus / alert-relay 审计文件
4. 导出：
   - `/metrics`
   - compose / systemd 日志
   - `nexus-alert-relay` 审计 JSONL

## 7. 排障入口

### 7.1 告警审计查询

```bash
python3 scripts/query_alert_audit.py --path /var/log/nexus/alert-relay/audit.jsonl --since 1h --summary alertname
python3 scripts/query_alert_audit.py --path /var/log/nexus/alert-relay --since 15m --contains dead-letter
```

### 7.2 Broker 恢复演练

```bash
./scripts/run_broker_failure_drill.sh rabbitmq runtime-worker
./scripts/run_broker_failure_drill.sh nats runtime-worker
./scripts/run_broker_failure_drill.sh redis_streams runtime-worker
```

### 7.3 配置复核

```bash
./scripts/check_deploy_config.sh
```
