# 一、NexusCode 总设计目标

## 1. 平台定位

`NexusCode` 是一个以“内容、计算、知识连接、发布”为核心的平台。  
它不是单一的 OJ、博客或笔记系统，而是一个统一承载以下能力的开发者平台：

- 在线评测系统 `NexusOJ`
- 双链知识与博客系统 `NexusBlog`
- 云函数与执行平台 `NexusFunction`
- 桌面双链笔记客户端 `NexusNote`
- 实时任务、内容发布、协作与扩展能力

平台核心目标是把“写作、编程、执行、发布、连接知识”整合成一套统一体验。

---

## 2. 愿景目标

NexusCode 希望形成如下闭环：

1. 用户在本地客户端或 Web 端创作内容
2. 内容可通过双链、标签、块引用形成知识网络
3. 内容可发布为博客、题解、文档、教程、说明页面
4. 用户可在平台中编写代码、运行函数、提交题目、查看实时反馈
5. 所有内容与执行结果都可以互相引用、关联、检索和再利用

平台最终希望成为一个面向开发者、学习者和创作者的统一知识与执行平台。

---

## 3. 核心设计原则

### 3.1 统一入口
平台对外应有统一访问入口，避免用户分别理解多个独立系统。

### 3.2 内容与执行并重
平台既要支持静态内容创作与发布，也要支持动态执行与实时反馈。

### 3.3 模块边界清晰
平台内部允许包含多个业务域，但每个业务域都应具备清晰边界。

### 3.4 先模块化单体，后可拆服务
第一阶段以模块化单体为主，优先降低复杂度；未来根据边界和负载演进为多服务架构。

### 3.5 Rust 统一核心能力
平台核心后端优先使用 Rust 实现，以获得更强的性能、类型安全、异步能力和系统级控制能力。

### 3.6 本地创作 + 平台发布
桌面客户端负责本地创作和编辑，平台负责同步、发布、渲染、分享和执行。

### 3.7 可扩展内容语法
平台支持标准 Markdown，并可扩展平台特有语法，如分栏、局部高亮、块引用增强等。

### 3.8 可观测、可维护、可持续演进
平台设计必须考虑日志、监控、配置、扩展性和长期维护成本。

---

## 4. 总体业务目标

### 4.1 OJ 目标
- 支持 ACM、Functional、EasyJudge、SPJ、Validator 等判题模式
- 支持实时提交反馈
- 支持题目录入、测试用例管理、提交记录、题解和题目内容管理

### 4.2 博客与内容目标
- 支持双链文档、块级结构、反向链接、内容发布
- 支持将笔记发布为博客页面
- 支持文档与 OJ 题目、题解、函数文档互相引用

### 4.3 云函数目标
- 支持定义函数、发布函数、调用函数、查看日志和结果
- 与 OJ 共用统一执行内核和资源控制机制

### 4.4 笔记客户端目标
- 提供类似 Obsidian 的双链笔记体验
- 支持平台自定义 Markdown 扩展
- 支持发布到 NexusCode 平台
- 支持未来与博客、题解、知识库联动

---

## 5. 阶段性目标

### 第一阶段
- 稳定 OJ 主链路
- 完成 Rust 平台边界设计
- 建立统一内容模型和统一网关模型

### 第二阶段
- 实现 Rust 版 `NexusGate`
- 建立内容系统基础
- 统一用户、权限、资源访问模型

### 第三阶段
- 抽象统一执行内核 `NexusRuntime`
- 迁移 OJ 执行能力
- 云函数开始接入

### 第四阶段
- Tauri 双链笔记客户端接入发布链路
- 内容系统与博客系统打通
- 知识网络、搜索、发布生态形成闭环
