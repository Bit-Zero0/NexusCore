# NexusGate 与 NexusRuntime 详细职责与交互文档

## 1. 文档目标

本文档用于明确 `NexusCode` 平台中两个核心模块的职责边界：

- `NexusGate`
- `NexusRuntime`

以及二者之间的交互方式、数据流转关系、状态变化、边界约束和未来演进方向。

本文档的核心目的，是避免后续设计中出现以下问题：

- 将所有逻辑都堆积到 `NexusGate`
- 将 `NexusRuntime` 误设计为业务系统
- 让平台入口层与执行层职责混乱
- 让 OJ、云函数、博客等业务在执行路径上相互耦合

---

## 2. 总体定位

`NexusGate` 和 `NexusRuntime` 是平台中最容易被混淆的两个核心模块，但它们本质上处于完全不同的层次。

### 2.1 NexusGate 的定位

`NexusGate` 是平台统一入口。

它面向：

- Web 前端
- 管理后台
- Tauri 客户端
- 外部调用方

它负责：

- 接收请求
- 鉴权
- 路由
- 聚合业务结果
- 管理实时连接
- 对外提供统一 API

一句话概括：

`NexusGate` 负责“平台如何对外说话”。

### 2.2 NexusRuntime 的定位

`NexusRuntime` 是平台统一执行内核。

它面向：

- OJ 判题任务
- 云函数执行任务
- 未来的脚本、自动化任务、编译任务等执行型任务

它负责：

- 编译
- 沙箱
- 执行
- 资源限制
- 超时控制
- 日志采集
- 执行结果回传

一句话概括：

`NexusRuntime` 负责“平台如何真正执行任务”。

---

## 3. 直观理解

可以用下面的类比来理解二者差异。

### 3.1 NexusGate 像前台与总控台

它负责：

- 接待用户
- 判断请求应该去哪里
- 返回结果给用户
- 维护与用户的实时通信

### 3.2 NexusRuntime 像后场引擎与执行车间

它负责：

- 实际把任务拿来处理
- 编译代码
- 在受控环境执行
- 收集执行结果
- 上报执行状态

### 3.3 它们的关系

- `NexusGate` 是入口
- `NexusRuntime` 是执行核心
- `NexusGate` 不应该亲自执行重任务
- `NexusRuntime` 不应该直接对外暴露用户 API

---

## 4. NexusGate 详细职责

## 4.1 统一入口职责

`NexusGate` 是整个 `NexusCode` 平台的唯一统一入口层之一。

其对外接口形式包括：

- HTTP API
- WebSocket
- 未来可扩展 gRPC

它需要承接的业务入口包括：

- OJ 提交、题目查询、提交记录
- 博客内容查询与发布
- 云函数调用入口
- 双链笔记发布入口
- 文件上传入口
- 平台用户接口

---

## 4.2 用户请求处理职责

`NexusGate` 需要负责：

- 请求解析
- 参数校验
- 权限校验
- 路由分发
- 统一错误码
- 统一返回结构

也就是说，`NexusGate` 不是简单转发器，而是平台面向客户端的协议层。

---

## 4.3 实时通信职责

`NexusGate` 需要统一管理：

- WebSocket 连接
- 订阅状态
- 心跳机制
- 断线恢复语义
- 消息推送格式

适用场景：

- OJ 提交状态
- 云函数执行日志
- 发布任务状态
- 集群状态更新

---

## 4.4 聚合职责

`NexusGate` 可以在平台边界做轻量聚合。

例如：

- 一次返回题目详情 + 用户提交状态
- 一次返回文档详情 + 发布状态
- 一次返回函数信息 + 最近调用记录

但聚合不意味着把业务逻辑搬到 `NexusGate`。

聚合仅限于：

- 跨模块查询结果组合
- 前端友好的结构组装
- 实时消息包装

---

## 4.5 轻量本地业务职责

并不是所有任务都必须进入 `NexusRuntime`。

`NexusGate` 可以直接完成的轻量任务包括：

- EasyJudge 这类纯规则判定
- 结构校验
- 配置归一化
- 草稿保存
- 内容发布预检查

设计原则是：

- 无需沙箱执行
- 不涉及不可信代码
- 不需要资源隔离
- 计算成本低

这类任务可以留在入口层或业务域内部完成。

---

## 4.6 不应由 NexusGate 承担的职责

`NexusGate` 不应负责：

- 编译用户代码
- 执行不可信代码
- 沙箱隔离
- 复杂任务调度
- 重计算型业务逻辑
- 长生命周期后台任务执行

否则会导致：

- 入口层过重
- 网关资源模型失控
- 高并发下稳定性下降
- 后续无法独立扩容

---

## 5. NexusRuntime 详细职责

## 5.1 执行内核职责

`NexusRuntime` 是平台统一执行内核。

它不是业务系统，不直接关心：

- 用户页面长什么样
- 题目如何展示
- 博客如何渲染
- 函数如何配置发布策略

它关心的是：

- 拿到什么任务
- 需要怎样执行
- 执行环境是什么
- 资源限制是什么
- 执行结果是什么

---

## 5.2 统一任务执行职责

`NexusRuntime` 应支持统一的任务模型。

任务至少应包含：

- 任务类型
- 执行语言/运行时
- 资源限制
- 输入数据
- 关联上下文
- 输出目标

典型任务类型：

- OJ_ACM_JUDGE
- OJ_FUNCTIONAL_JUDGE
- OJ_SPJ_JUDGE
- FUNCTION_INVOKE

---

## 5.3 资源与隔离职责

`NexusRuntime` 应统一负责：

- CPU 限制
- 内存限制
- 时间限制
- 进程数量限制
- 文件系统隔离
- 网络权限限制
- 沙箱环境准备

这也是它与 `NexusGate` 的根本区别之一。

---

## 5.4 日志与执行结果职责

`NexusRuntime` 应收集并规范化：

- 编译日志
- 运行错误日志
- 标准输出
- 标准错误
- 用时
- 内存
- 退出码
- 任务状态

并将这些结果上报给业务模块或事件系统。

---

## 5.5 重试与失败处理职责

`NexusRuntime` 应负责执行侧的失败语义：

- 是否可重试
- 重试次数
- 延迟重试
- 死信处理
- 执行器异常恢复

这类逻辑不应由前端接口层决定。

---

## 5.6 不应由 NexusRuntime 承担的职责

`NexusRuntime` 不应负责：

- 题目编辑
- 文档编辑
- 用户权限判断
- 博客页面渲染
- OJ 页面结构组织
- 平台 UI 需要的接口聚合

它是执行引擎，不是平台业务中台。

---

## 6. 二者的边界差异总结

| 维度 | NexusGate | NexusRuntime |
|---|---|---|
| 核心定位 | 平台入口 | 执行内核 |
| 面向对象 | 用户、客户端、外部调用方 | 执行任务、工作节点 |
| 处理内容 | 请求、响应、鉴权、订阅 | 编译、运行、隔离、结果 |
| 协议 | HTTP / WS / 未来 gRPC | MQ / 内部任务协议 / 事件 |
| 资源模型 | 高并发轻逻辑 | 低并发重执行 |
| 安全重点 | 鉴权与输入控制 | 沙箱与资源隔离 |
| 伸缩方式 | 按连接数与请求数扩容 | 按计算资源扩容 |
| 是否直接面对前端 | 是 | 否 |

---

## 7. 典型交互流程

## 7.1 OJ 提交流程

### 流程

1. 前端向 `NexusGate` 发起提交请求
2. `NexusGate` 完成：
   - 鉴权
   - 参数校验
   - 题目配置查询
   - 创建提交记录
   - 生成任务描述
3. `NexusGate` 将任务投递给任务系统
4. `NexusRuntime` 消费任务
5. `NexusRuntime` 编译并执行
6. `NexusRuntime` 产出执行结果
7. 结果回写业务存储或发布事件
8. `NexusGate` 通过 WebSocket 向前端推送状态

### 核心原则

- `NexusGate` 负责“提交”
- `NexusRuntime` 负责“执行”

---

## 7.2 EasyJudge 提交流程

### 流程

1. 前端向 `NexusGate` 提交简单题答案
2. `NexusGate` 查询题目结构
3. `NexusGate` 在轻量逻辑中直接完成判定
4. `NexusGate` 直接保存结果并回推状态

### 核心原则

由于不涉及不可信代码与沙箱资源，因此这类任务不必进入 `NexusRuntime`。

---

## 7.3 云函数调用流程

### 流程

1. 客户端调用函数 API 到 `NexusGate`
2. `NexusGate` 完成：
   - 身份校验
   - 函数元数据校验
   - 调用参数校验
   - 创建 invocation 记录
3. `NexusGate` 生成函数执行任务
4. `NexusRuntime` 执行函数
5. `NexusRuntime` 产生日志和结果
6. `NexusGate` 将结果返回给用户或实时推送

### 核心原则

函数调用属于业务入口 + 执行内核协作，不应由单一模块独占全部逻辑。

---

## 8. 数据交互模型

## 8.1 从 Gate 到 Runtime 的数据

`NexusGate` 不应将领域数据库对象原样传给 `NexusRuntime`。

应传递的是规范化的“执行任务描述”：

- task_id
- task_type
- language/runtime
- source_code
- input / testcase refs
- judge config
- resource limits
- callback / event metadata

这样做的原因：

- 降低耦合
- 便于未来服务化
- 保持执行内核的通用性

---

## 8.2 从 Runtime 返回 Gate/业务域的数据

`NexusRuntime` 不应直接返回面向 UI 的复杂结构。

应返回：

- 执行状态
- 编译结果
- 运行结果
- 日志
- 每个阶段的原始结果
- 资源使用指标

然后由业务模块或 `NexusGate` 再整理成前端可展示结构。

---

## 9. 实时消息交互

`NexusGate` 应是平台实时消息出口。

推荐模式：

1. `NexusRuntime` 发布执行状态事件
2. OJ / Function 等业务域消费并更新状态
3. `NexusGate` 订阅或获取这些状态变化
4. `NexusGate` 向客户端推送统一事件格式

这样可以确保：

- 前端不直接理解执行内核语义
- 不同业务域可共享同一实时推送模型

---

## 10. 资源扩容逻辑

## 10.1 NexusGate 的扩容方式

扩容指标：

- HTTP QPS
- WebSocket 连接数
- API 延迟
- 内存占用

它适合：

- 多副本无状态扩容
- 部署在 API 网关层或入口层

## 10.2 NexusRuntime 的扩容方式

扩容指标：

- CPU 核数
- 内存
- 队列堆积长度
- 执行耗时
- 节点负载

它适合：

- 按执行能力水平扩容
- 按 `fast / normal / heavy` 或不同 runtime 能力分池

### 这也是二者必须拆清的现实原因

网关与执行器的资源形态完全不同，混在一个服务中长期不可维护。

---

## 11. 未来服务化演进

虽然第一阶段推荐模块化单体，但从边界上看：

### 11.1 NexusGate 很适合先独立

因为：

- 对外 API 边界天然明确
- 伸缩方式独立
- 与前端和客户端关系最强

### 11.2 NexusRuntime 也非常适合独立

因为：

- 资源模型与入口层完全不同
- 执行负载与业务逻辑不同
- 需要独立按能力扩容

因此，在未来演进中，`Gate` 和 `Runtime` 很可能是最早拆开的两个核心模块。

---

## 12. 架构设计约束

为了确保边界长期有效，必须遵守以下约束：

### 12.1 Gate 不直接执行不可信代码

### 12.2 Runtime 不直接处理前端协议

### 12.3 Gate 与 Runtime 之间只传规范化任务协议

### 12.4 Runtime 不依赖业务 UI 模型

### 12.5 业务模块不把 Runtime 当数据库或页面服务使用

---

## 13. 结论

`NexusGate` 和 `NexusRuntime` 的关系可以总结为：

- `NexusGate` 负责“入口、接口、订阅、聚合”
- `NexusRuntime` 负责“执行、隔离、资源控制、结果产出”

它们共同构成 `NexusCode` 平台中最重要的两类平台能力：

- 平台统一入口能力
- 平台统一执行能力

只有把这两层彻底拆清，`NexusCode` 后续的：

- OJ
- 博客
- 云函数
- 笔记发布

才能在同一个平台中长期共存，而不相互污染。
