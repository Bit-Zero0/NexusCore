# NexusCode 总体架构图与部署演进文档

## 1. 文档目标

本文档用于描述 `NexusCode` 的总体架构形态、核心模块关系、部署视图以及从“模块化单体”逐步演进到“可拆分平台”的路线。

本文档回答的问题包括：

- `NexusCode` 平台整体由哪些部分组成
- 各部分在逻辑上如何连接
- 为什么第一阶段适合模块化单体
- 未来应如何逐步拆分与独立部署
- 不同阶段的部署目标、边界和风险是什么

本文档是平台级架构文档，不涉及具体代码实现。

---

## 2. 平台总体定位

`NexusCode` 是一个统一的开发者平台，而不是多个无关系统的简单拼接。

平台同时承载以下业务方向：

- `NexusOJ`：在线评测
- `NexusBlog`：博客与知识发布
- `NexusFunction`：云函数与执行平台
- `NexusNote`：Tauri 双链笔记客户端

这些业务虽然面向不同场景，但共享以下平台能力：

- 用户与权限
- 内容管理
- 实时推送
- 执行调度
- 统一网关
- 存储与资源管理

因此，`NexusCode` 的总体架构应体现“统一平台，多个业务域，共享底层能力”的思路。

## 3.2 职责解释

### Client Layer

包括：

- 用户 Web 前端
- 管理后台
- Tauri 笔记客户端
- 外部 API 调用者

### NexusGate

平台入口层，统一承接：

- HTTP API
- WebSocket
- 鉴权入口
- 前端接口聚合
- 实时状态推送

### Business Layer

业务域层，承载：

- 用户与权限
- OJ
- 内容与博客
- 云函数
- 笔记同步

### Platform Services

平台共享能力层，承载：

- 实时消息能力
- 事件总线抽象
- 存储访问抽象
- 搜索抽象

### NexusRuntime

统一执行内核，负责：

- 不可信代码执行
- 沙箱与资源限制
- 编译与运行
- 执行状态上报

### Infrastructure Layer

底层基础设施：

- PostgreSQL
- Redis
- MQ
- 对象存储
- 搜索引擎

## 4. 第一阶段总体架构

## 4.1 核心目标

第一阶段的目标不是“服务数量多”，而是：

- 模块边界清晰
- 业务可快速迭代
- 部署简单
- 联调简单
- 为未来拆分留好边界

因此，第一阶段推荐采用：

**模块化单体 + 统一 workspace + 单主程序部署**

---

## 4.2 第一阶段架构图

```text
┌─────────────────────────────────────────┐
│      Web / Admin / Tauri / API Client   │
└─────────────────────┬───────────────────┘
                      │
                      ▼

│           nexus-app (single process)    │
│                                         │
│  - nexus-gateway                        │
│  - nexus-auth                           │

│  - nexus-shared                         │

                      │
                      ▼
┌─────────────────────────────────────────┐
│ PostgreSQL / Redis / MQ / ObjectStore   │
└─────────────────────────────────────────┘
```

---

## 4.3 为什么第一阶段适合单体部署

原因如下：

### 4.3.1 产品仍处于快速演进期

平台的业务边界还在逐步定型。过早微服务化会让演进成本大幅上升。

### 4.3.2 当前最重要的是边界验证

先要验证的是：

- `Gateway` 是否该这样定义
- `Runtime` 是否足够通用
- `Content` 和 `OJ` 是否边界合理

而不是先做网络化拆分。

### 4.3.3 降低复杂度

单体部署更适合当前阶段的原因：

- 本地联调简单
- 日志集中
- 配置集中
- 调试成本低
- 不需要先解决服务发现、跨服务鉴权、链路追踪等问题

## 5. 第一阶段部署方案

## 5.1 部署目标

提供一个可以支撑核心功能的统一平台部署实例。

### 需要支撑的核心功能

- OJ 提交与结果流
- 博客/内容基础接口
- 云函数基础定义与调用接口
- 客户端登录与发布接口

---

## 5.2 第一阶段部署组成

至少包括：

- `nexus-app` 一个主程序
- PostgreSQL 一个实例
- Redis 一个实例
- RabbitMQ 或其他 MQ 一个实例
- 对象存储一个实例或兼容服务

---

## 5.3 第一阶段部署特点

- 简单
- 集中
- 容易迁移
- 容易观察问题

但缺点也明确：

- 入口与执行进程资源竞争
- 故障隔离能力有限
- 水平扩展粒度较粗

---

## 6. 第二阶段演进：逻辑单体，物理分离执行节点

## 6.1 演进触发条件

当出现以下情况时，应进入第二阶段：

- OJ 判题和函数执行明显占用大量 CPU / 内存
- API 服务因执行任务受影响
- 执行节点需要按能力分层扩容

---

## 6.2 第二阶段目标

将执行型模块从入口型模块中物理分离，但业务边界仍保持清晰统一。

### 演进方式

- `nexus-gateway + 业务模块` 仍可集中部署
- `nexus-runtime` 独立部署为执行节点组

---

## 6.3 第二阶段架构图

```text
┌────────────────────────────────────────────┐
│      Web / Admin / Tauri / API Client      │
└──────────────────────┬─────────────────────┘
                       │
                       ▼
┌────────────────────────────────────────────┐
│                Nexus Platform              │
│  nexus-gateway + auth + content + oj +     │
│  function + event + storage abstractions   │
└──────────────────────┬─────────────────────┘
                       │
             ┌─────────┴─────────┐
             ▼                   ▼
┌──────────────────────┐  ┌──────────────────────┐
│   Runtime Worker A   │  │   Runtime Worker B   │
│   fast / normal      │  │   heavy / special    │
└──────────────────────┘  └──────────────────────┘
             │                   │
             └─────────┬─────────┘
                       ▼
┌────────────────────────────────────────────┐
│ PostgreSQL / Redis / MQ / Object Storage   │
└────────────────────────────────────────────┘
```

---

## 6.4 第二阶段收益

- 网关与执行器资源隔离
- 执行节点可独立扩容
- 可以更自然支持 `fast / normal / heavy`
- 更适合 OJ 与云函数共享执行内核

---

## 7. 第三阶段演进：平台入口独立

## 7.1 演进触发条件

当出现以下信号时，`NexusGate` 应考虑独立：

- WebSocket 连接显著增加
- 外部接口复杂度提高
- 入口层需要高频发布
- 前端和客户端依赖强烈集中于统一 API 层

---

## 7.2 第三阶段目标

将 `NexusGate` 独立为对外统一入口服务。

同时业务模块可选择仍在一个平台服务中，也可逐步拆开。

---

- 统一入口更清晰
- 前端和客户端协作模型更稳定
- 入口层可单独扩容、独立上线
- 更适合后续引入 gRPC、SSE、统一 API 版本管理

---

## 8. 第四阶段演进：业务域逐步独立

## 8.1 演进触发条件

只有当业务域成熟、复杂度稳定后，才考虑进一步拆分业务模块。

可能的触发条件：

- 博客/内容系统发展为完整内容平台
- OJ 比赛与评测业务规模明显扩大
- 云函数平台有独立的高频调用与部署节奏
- 团队协作需要明确 ownership

---

## 8.2 第四阶段目标

将部分业务域独立为服务，但保留统一平台能力。

潜在拆分方向：

- `nexus-content-service`
- `nexus-oj-service`
- `nexus-function-service`

---

## 8.4 第四阶段风险

业务域拆分过早，可能带来：

- 跨服务事务复杂度上升
- 联调成本提高
- 权限边界复杂化
- 调试难度上升

因此，业务域拆分必须晚于边界稳定。

---

## 9. 第五阶段演进：内容平台与笔记发布闭环

## 9.1 演进目标

完成 `NexusNote` 与平台的完整闭环：

- 本地编辑
- 双链与扩展 Markdown
- 发布到平台
- 博客访问
- 反向链接与知识图谱

### 此时平台结构会更偏向三大中心

- 内容中心
- 执行中心
- 平台入口中心

---

## 10. 部署演进路线总览

## 10.1 阶段总结表

| 阶段 | 形态 | 核心目标 | 是否适合当前阶段 |
|---|---|---|---|
| Phase 1 | 模块化单体 | 快速迭代、边界验证 | 是 |
| Phase 2 | 入口与执行分离 | 隔离重任务资源 | 很可能很快需要 |
| Phase 3 | Gate 独立 | 统一入口稳定化 | 中期需要 |
| Phase 4 | 业务域逐步独立 | 大规模平台化 | 后期再做 |
| Phase 5 | 完整平台生态 | 内容、执行、发布闭环 | 长期目标 |

---

## 10.2 推荐现实路线

对 `NexusCode` 来说，最推荐的演进次序是：

1. 先做 Rust workspace 与模块化单体
2. 再把 `Runtime` 独立部署
3. 再把 `Gate` 独立部署
4. 业务域成熟后再逐步拆分

而不是：

1. 一开始就做一堆微服务

---

## 11. 为什么 Runtime 往往先独立

因为它的资源模型与 `Gateway` 差异最大。

### Gateway 的资源特征

- 更多网络连接
- 更多请求并发
- 较轻逻辑
- 更依赖低延迟

### Runtime 的资源特征

- 更高 CPU 占用
- 更高内存占用
- 执行时间更长
- 更依赖隔离环境

两者混跑时，最容易互相影响，因此 `Runtime` 是最适合最早物理分离的模块之一。

---

## 12. 为什么 Gateway 之后也适合独立

随着平台扩张，`Gateway` 会越来越像：

- 平台统一 API 层
- Realtime Hub
- BFF
- 统一鉴权入口

这类模块天然更适合独立：

- 版本管理更清晰
- 前端协作更稳定
- 可单独做 API 策略与限流

---

## 13. 部署环境建议

## 13.1 开发环境

推荐：

- 单机 Docker 基础设施
- 单进程 `nexus-app`
- 独立前端开发服务

### 目标

- 快速本地联调
- 最少基础设施成本

---

## 13.2 测试环境

推荐：

- 单独 PostgreSQL / Redis / MQ
- `nexus-app` 或 `Gate + Runtime` 两段部署
- 基础监控

### 目标

- 验证真实部署链路
- 验证消息流与执行流

---

## 13.3 生产环境

早期推荐：

- `NexusGate/Core` 一组
- `NexusRuntime` 独立一组
- PostgreSQL 高可用或可备份方案
- Redis 独立部署
- MQ 独立部署
- 对象存储外置

---

## 14. 可观测性要求

无论在哪个阶段，以下能力都应尽早规划：

- 结构化日志
- 指标采集
- 任务链路追踪
- 错误聚合
- 节点健康检查

特别是：

- `Gateway` 关注请求级指标
- `Runtime` 关注任务级指标

这两类指标不应混在一起理解。

---

## 15. 架构演进约束

为了确保平台演进可控，必须遵守以下原则：

### 15.1 不以“服务数量”为目标

目标是边界合理，不是服务越多越先进。

### 15.2 先定义接口和事件，再考虑拆分

如果接口和事件模型没定，拆服务只会更混乱。

### 15.3 执行层必须与入口层长期解耦

这是平台能否承载 OJ 和云函数的关键。

### 15.4 内容系统必须是一级能力

博客、笔记发布、题解等都依赖它，不能把内容系统当附属模块。

---

## 16. 最终架构目标

从长期看，`NexusCode` 理想形态应是：

- 一个统一平台入口 `NexusGate`
- 一个统一执行内核 `NexusRuntime`
- 多个相互独立但共享基础能力的业务域
- 一个可被 Tauri 客户端和 Web 客户端共同使用的平台
- 一套内容、执行、发布、实时反馈统一协作的生态

也就是说，平台最终不是“一个 OJ 加一个博客”，而是：

**一个统一平台，多个能力中心，多个业务域协同运行。**

---

## 17. 结论

`NexusCode` 的总体架构应采用分阶段演进策略：

1. 先模块化单体，验证边界
2. 再独立执行内核，隔离重资源任务
3. 再独立统一入口，稳定前端与客户端接口
4. 最后按成熟度拆分业务域

这条路线的优势在于：

- 与当前项目阶段匹配
- 可以保护现有成果
- 不会过早引入微服务复杂度
- 能为 OJ、博客、云函数和笔记平台统一演进留出空间

这是 `NexusCode` 当前最现实、最稳妥、也最具有长期价值的架构路线。
