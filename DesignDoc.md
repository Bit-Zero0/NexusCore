




# 三、NexusCode 总设计文档

## 1. 总体架构定位

NexusCode 采用“模块化单体优先、未来可服务化”的总体路线。  
第一阶段以内聚的 Rust workspace 为核心，通过清晰模块边界实现低耦合设计；后续按边界逐步独立部署。

---

## 2. 架构分层

## 2.1 客户端层
包括：
- Web 前端
- 管理后台
- Tauri 双链笔记客户端

职责：
- 用户交互
- 内容编辑
- 提交发起
- 状态展示
- 发布操作

---

## 2.2 平台入口层
由 `nexus-gate` 承担。

职责：
- HTTP API
- WebSocket 实时推送
- 统一鉴权入口
- 前端聚合接口
- 文件上传入口
- 平台路由入口

不负责：
- 执行沙箱逻辑
- 深度业务计算
- 领域内部存储细节

---

## 2.3 业务域层
包括：

- `nexus-auth`
- `nexus-content`
- `nexus-oj`
- `nexus-function`

每个模块拥有：
- 自己的领域模型
- 自己的 service
- 自己的 repository
- 自己的事件

模块之间通过明确接口协作，不直接越权访问彼此内部实现。

---

## 2.4 执行内核层
由 `nexus-runtime` 承担。

职责：
- 沙箱
- 编译
- 执行
- 资源限制
- 超时控制
- 日志采集
- 执行结果回传

它是 OJ 和云函数共享的执行核心。

---

## 2.5 基础设施层
包括：
- PostgreSQL
- Redis
- MQ
- Object Storage
- Search
- Metrics / Logs / Trace

---

## 3. 模块设计

## 3.1 `nexus-gate`

### 职责
- 平台统一 API 入口
- 统一 WebSocket 管理
- 统一认证拦截
- 统一响应格式
- 与前端交互的 BFF

### 提供能力
- OJ 接口
- 博客接口
- 文档接口
- 云函数调用接口
- 提交实时更新
- 集群状态查询

### 不承担
- 判题执行
- 内容 AST 存储细节
- 函数运行时实现

---

## 3.2 `nexus-auth`

### 职责
- 用户管理
- 登录态
- Token/JWT
- 权限控制
- 角色模型

### 目标
为所有业务域提供统一身份能力。

---

## 3.3 `nexus-content`

### 职责
- 文档模型
- Markdown 解析
- 双链与反链
- 发布状态
- 资源引用
- 渲染准备数据

### 核心对象
- `Document`
- `Block`
- `Link`
- `Backlink`
- `Asset`
- `PublishRecord`
- `RenderProfile`

---

## 3.4 `nexus-oj`

### 职责
- 题目管理
- 测试用例管理
- 提交记录
- 判题任务生成
- 集群路由策略
- 结果聚合

### 支持模式
- ACM
- Functional
- EasyJudge
- Validator
- SPJ

### 特殊说明
- EasyJudge 可由平台快速本地判定
- 其他计算型任务通常进入统一任务链路

---

## 3.5 `nexus-function`

### 职责
- 函数定义
- 版本管理
- 调用管理
- 执行记录
- 日志查看

### 与 Runtime 的关系
函数执行本身不在本模块内部完成，而是投递给 `nexus-runtime`。

---

## 3.6 `nexus-runtime`

### 职责
- 拉取任务
- 准备执行环境
- 应用资源限制
- 编译或解释执行
- 收集输出
- 回传状态与结果

### 支持任务类型
- OJ 判题任务
- 云函数执行任务
- 未来可扩展执行型任务

### 设计价值
把“执行”从具体业务域抽离，避免 OJ 和云函数各自维护一套执行系统。

---

## 4. 模块依赖方向

推荐依赖方向：

- `nexus-app` 依赖所有业务模块
- `nexus-gate` 依赖业务模块公开接口
- 业务模块依赖 `nexus-shared`
- 业务模块不互相直接依赖内部 repository
- `nexus-runtime` 只依赖任务协议和共享类型

禁止方向：
- `gateway` 直接写业务 SQL
- `oj` 直接读 `content` 内部表
- `function` 直接操纵 `auth` 存储
- 任意模块跨边界修改别的模块内部状态

---

## 5. 数据存储设计

## 5.1 PostgreSQL
用于存储核心持久化数据：
- 用户
- 题目
- 测试用例
- 提交
- 文档
- 发布记录
- 函数定义
- 版本记录

建议按域分 schema：
- `auth`
- `oj`
- `content`
- `function`

---

## 5.2 Redis
用于：
- 会话
- 缓存
- 实时状态
- 节点心跳
- WebSocket 辅助状态

---

## 5.3 MQ
用于：
- 判题任务
- 函数执行任务
- 事件广播
- 重试与死信

---

## 5.4 对象存储
用于：
- 图片
- 附件
- 发布资源
- 题目资源
- 构建产物或中间文件

---

## 6. 内容系统设计原则

### 6.1 内容不只是字符串
文档必须存在结构化表达能力，最终要有 AST 概念。

### 6.2 扩展语法不应依赖脆弱正则
平台扩展语法应基于解析器和中间表示层设计。

### 6.3 发布是内容生命周期的一部分
文档有：
- 草稿
- 已发布
- 已更新待发布
- 已归档
等生命周期。

---

## 7. 执行系统设计原则

### 7.1 执行是平台能力，不是 OJ 私有能力
OJ、云函数、未来脚本执行都应复用统一运行内核。

### 7.2 执行任务必须具备统一状态机
建议统一状态：
- pending
- queued
- running
- success
- failed
- timeout
- cancelled

### 7.3 必须支持重试与死信
尤其是异步执行任务，必须具备：
- retry
- delay retry
- dead letter
- reconnect
- result recovery

---

## 8. 实时系统设计

平台需要统一实时能力：
- WebSocket 连接
- 订阅确认
- 心跳
- 断线恢复
- 事件类型规范
- 多业务域消息统一语义

适用场景：
- OJ 提交结果
- 函数执行状态
- 发布任务状态
- 集群监控状态

---

## 9. 客户端设计原则

### Web
适合：
- OJ
- 博客访问
- 管理后台
- 发布管理
- 集群监控

### Tauri Note
适合：
- 本地创作
- 双链管理
- 平台扩展语法编辑
- 发布到博客
- 知识沉淀

---

## 10. 演进路线

### Phase 1
- 统一平台目标
- 统一领域边界
- 稳住现有 OJ 功能模型

### Phase 2
- Rust 版 `nexus-gate`
- Rust workspace 搭建
- 公共模型与配置体系建立

### Phase 3
- Rust 版 `nexus-oj`
- Rust 版 `nexus-runtime`
- OJ 执行链迁移

### Phase 4
- `nexus-content`
- `nexus-note-sync`
- 博客与笔记发布链打通

### Phase 5
- `nexus-function`
- 统一执行模型扩展到云函数

---

## 11. 风险与约束

### 风险
- 全量重写成本高
- 边界定义不清会导致后续混乱
- 内容模型过晚定义会影响博客与笔记系统
- 执行内核与业务域耦合过深会影响扩展

### 约束
- 第一阶段应避免微服务过度设计
- 必须以模块化单体为主
- 必须优先定义接口、事件和模型边界
- 必须保证 OJ 主链路不被无节制重构拖垮

---

## 12. 最终结论

NexusCode 不应被设计成“多个毫无关系的小系统集合”，而应被设计成：

- 一个统一平台
- 多个业务域
- 一个统一平台入口
- 一个统一执行内核
- 一个统一内容模型
- 多客户端协同

这会让 OJ、博客、云函数、双链笔记彼此增强，而不是彼此割裂。
