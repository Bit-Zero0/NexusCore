# P1 阶段收口清单

## 1. 文档目的

本文档用于正式收口当前“多 Broker 平台底座”的 P1 阶段。

P1 的目标不是把平台做成完整运维产品，而是把平台推进到下面这个标准：

`控制面可直接用、运维可直接排障、语义不再模糊。`

当这三点成立时，P1 就应该停止继续扩张，转入下一阶段的统一 Job 平台接口设计。

---

## 2. P1 完成标准

P1 是否完成，以以下 4 条为准：

1. 管理视图可被控制面直接消费
2. 常见运维排障路径不需要人工拼日志和猜状态
3. Broker 关键语义已经被显式化且不再模糊
4. 剩余未做事项不再阻塞统一 Job 平台接口设计

---

## 3. 已完成

## 3.1 控制面可直接使用的管理视图已落地

当前已经具备：

- `GET /api/v1/runtime/management/broker`
- `GET /api/v1/runtime/management/runbooks`

相关文档与实现：

- [Runtime_Gateway_运维_API_契约文档.md](/home/fmy/NexusCore/Runtime_Gateway_运维_API_契约文档.md)
- [crates/nexus-runtime/src/router.rs](/home/fmy/NexusCore/crates/nexus-runtime/src/router.rs)
- [crates/nexus-runtime/src/executor.rs](/home/fmy/NexusCore/crates/nexus-runtime/src/executor.rs)

当前管理视图已经包含：

- broker observability profile
- queue stats
- worker groups
- dead letters
- replay history
- 分页与过滤
- 独立总数
- health 状态
- degradation_reasons
- alerts 摘要
- recommended_action
- action_kind
- runbook_ref
- runbook 元信息

这意味着控制面已经不需要再自己拼：

- broker 状态
- 失败来源
- 推荐动作
- runbook 跳转信息

---

## 3.2 运维排障路径已经足够直接

当前已经具备：

- `queue / lane / group` 过滤
- `task_id / delivery_id` 检索
- dead-letter 列表
- replay 历史
- broker health / degradation 摘要
- runbook catalog

这意味着运维在面对一条异常任务时，已经可以较直接地回答：

- 它落在哪个 queue / lane
- 它是否进入 dead-letter
- 它是否已经 replay 过
- 当前 broker 是否处于恢复窗口
- 当前 degraded 的原因是什么
- 应该查看哪份 runbook

---

## 3.3 Broker 关键语义已经显式化

当前已经明确并落地的语义包括：

- `enqueue / reserve / ack / retry / reject`
- dead-letter / replay
- reclaim / recovery
- required capabilities
- enhanced capabilities
- degraded 的结构化原因
- recommended action 的结构化表示

这意味着后续统一 Job 平台接口不需要再重新解释：

- 什么叫 broker 健康
- 什么叫恢复窗口
- 什么叫持续失败
- 什么动作建议由控制面直接展示

---

## 3.4 Broker 极端场景验证已经达到 P1 需要的强度

当前 broker contract tests 已覆盖：

- 重复投递收敛
- 延迟重试边界
- 多 route 轮询与公平性
- dead-letter replay 一致性
- crash reclaim / recovery

相关实现：

- [crates/nexus-runtime/tests/support/broker_contract.rs](/home/fmy/NexusCore/crates/nexus-runtime/tests/support/broker_contract.rs)

这意味着 P1 阶段关心的“语义是否稳定”已经有自动化验证支撑。

---

## 3.5 P0 与 P1 已形成连续闭环

P0 解决了：

- 观测
- 告警
- 演练
- 配置收口
- CI 回归

P1 在此基础上解决了：

- 管理视图
- 排障检索
- 结构化 health / degradation
- 推荐动作与 runbook 跳转

这意味着当前平台已经不只是“能跑”，而是“能解释、能排障、能继续被平台化复用”。

---

## 4. 明确不再属于 P1 的内容

为了避免 P1 无限扩张，下面这些内容不再继续算作 P1 必做项：

- 更复杂的运维后台 UI
- 更重的全文检索/历史分析系统
- 更复杂的运维聚合报表
- 更完整的多租户控制面
- 更深入的长周期容量/性能分析系统
- 统一 Job 平台接口本身

这些内容应归入后续阶段，而不是继续留在 P1 范围里。

---

## 5. 进入 P2 前还允许保留的“小尾巴”

以下事项即使暂时不补，也不阻塞进入 P2：

- replay/dead-letter 更复杂的时间窗口聚合
- 更丰富的 runbook 章节映射
- 更细粒度的排障历史摘要
- 更复杂的控制面分页/排序能力

这些属于“增强项”，不是 P1 是否完成的判断门槛。

---

## 6. P1 当前结论

按本文档定义的完成标准判断：

`P1 已达到收口条件。`

原因是：

- 控制面已经可以直接消费管理视图
- 运维已经可以较直接地进行排障
- Broker 关键语义已经结构化且不再模糊
- 剩余问题已经不再阻塞统一 Job 平台接口设计

所以继续在 P1 上投入的边际收益会明显下降。

更合理的下一步应该是：

`正式进入 P2：统一 Job 平台接口设计与第一批业务接入验证。`

---

## 7. 建议的后续动作

建议按这个顺序继续：

1. 将本文档作为 P1 正式收口记录
2. 在阶段总结中注明 P1 已完成
3. 开始输出统一 Job 平台接口设计草图
4. 以 OJ 为第一批接入对象验证 Job 抽象
5. 再为博客模块和云函数模块提供统一接入面

---

## 8. 一句话结论

P1 的目标不是把平台做成完整运维产品。

P1 的目标是把平台做成：

`控制面能直接使用、运维能直接排障、平台语义足够稳定清晰。`

按这个标准，当前 P1 已经可以正式收住。
